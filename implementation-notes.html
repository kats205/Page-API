Phase 2 - Webhook Service

Timestamp: 2026-05-30 Asia/Bangkok

Design decisions: Replaced the comment-only normalizer with FacebookEventNormalizer that supports Page feed comments and Messenger text messages. The webhook publishes shared RawEventMessage JSON to raw_events.

Deviations: Unsupported webhook payload shapes are accepted and logged as ignored instead of failing the request, so Facebook does not retry irrelevant events.

Tradeoffs: The raw Facebook sub-payload is stored as compact JSON in payload_json rather than expanded into many optional columns. This preserves audit data without overfitting to Facebook payload variants.

Phase 3 - Core Service

Timestamp: 2026-05-30 Asia/Bangkok

Design decisions: Replaced the template Core Service with a Kafka consumer for raw_events, a PostgreSQL-backed state repository, a deterministic decision engine, and optional Gemini analysis. Gemini is used only when configured; otherwise the tested rule fallback handles the assignment sample cases.

Deviations: Bounded concurrency is currently conservative: the consumer processes and commits one event at a time to avoid offset commit bugs. This is safer for correctness; it can be scaled later by running multiple instances in the same consumer group.

Tradeoffs: Parallel in-process workers would improve throughput but make offset commits and DB transaction boundaries riskier. The current approach prioritizes no lost/duplicate events for the assignment demo.

Open questions: Confirm whether you want a real Gemini API key configured in appsettings, user-secrets, or environment variables.

Verification: Realtime tests passed 6/6. Core JSON config parses. dotnet build Page-API.sln passed with 0 warnings and 0 errors.

Phase status: PASS. Core Service now consumes raw_events, stores state, classifies/rules events, and publishes reply_commands.

Phase 4 - Backend API Command Consumer

Phase 5 - Retry Service

Timestamp: 2026-05-30 Asia/Bangkok

Design decisions: Replaced the template worker with a Kafka retry worker that consumes send_failed, applies the shared retry planner, and publishes either send_retry or dead_letter.

Deviations: Retry delay is implemented in the worker with Task.Delay. This matches the assignment examples of 1s, 2s, 4s and keeps local demo behavior visible.

Tradeoffs: A DB-backed delayed queue is better for very long delays or service restarts. Kafka plus short in-worker delays is simpler and sufficient for the required N=3 exponential backoff demo.

Open questions: Confirm whether you want retry attempts persisted by retry-service too, or whether backend-api's failed_commands table is enough as the audit log.

Verification: Realtime tests passed 7/7 using dotnet run --no-restore. dotnet build Page-API.sln --no-restore passed with 0 warnings and 0 errors.

Phase status: PASS. Retry Service now routes send_failed to send_retry or dead_letter.
